1. Introduction
1.1 Purpose
This Technical Requirements Document (TRD) defines the engineering specifications, system design, API contracts, data schemas, and integration requirements for TED (Tech Express Desk) — an AI-powered IT support kiosk. This document covers the software, AI, and ITSM layers only (Phase 1). Hardware/CV components are addressed in a separate TRD.

1.2 Scope
Authentication service (SSO + Badge RFID)
AI Diagnostic Engine (error matching, OCR, fix generation)
RAG pipeline for KB and past ticket lookup
Freshworks ITSM integration (ticket creation, KB, routing)
Kiosk UI (React touchscreen application)
FastAPI backend (all service endpoints)
Data storage (PostgreSQL + Redis)
Security, monitoring, and audit logging

1.3 Assumptions
Freshworks is the organisation's ITSM platform with API access enabled
SSO is available via existing corporate identity provider (Azure AD / Okta)
Kiosk hardware has network connectivity (Ethernet or Wi-Fi)
All services deployed on-device (edge) with cloud API calls for AI and Freshworks

2.2 Request Flow
Employee authenticates at kiosk (badge/SSO) → Auth Service validates → JWT issued
Employee connects device or selects issue → UI sends request to FastAPI
FastAPI triggers Diagnostic Engine → error DB match + OCR pipeline
If match found: fix steps returned to UI → Employee attempts resolution
If no match: RAG pipeline queries vector DB (KB + past tickets)
If unresolved: Freshworks ticket auto-created via ITSM Connector
Outcome logged to PostgreSQL → session closed → audit entry written

2.3 Deployment Topology
All services run as Docker containers via Docker Compose on kiosk device
Freshworks API calls made over HTTPS to cloud
LLM inference: cloud API (OpenAI / Groq / Azure OpenAI) — not on-device
Vector DB: cloud-hosted (Pinecone) or local (pgvector extension on PostgreSQL)
Redis runs locally on kiosk device for session caching

4. AI Diagnostic Engine
4.1 Overview
The AI Diagnostic Engine is the core intelligence of TED. It receives structured input (device info, error text, screenshots) and returns a diagnosis, confidence score, and step-by-step fix recommendation. It operates as a hybrid: rule-based error matching first (fast, deterministic), then LLM-based analysis for unmatched cases.

4.2 Input Schema
  POST /api/diagnose
  {
    "session_id": "string",
    "device_info": { "os": "string", "model": "string", "serial": "string" },
    "error_text": "string",          // from OCR or manual input
    "screenshot_b64": "string|null", // base64 encoded PNG
    "recording_url": "string|null",  // path to screen recording file
    "issue_description": "string"    // employee free-text description
  }

4.3 Output Schema
  {
    "diagnosis": "string",
    "confidence": 0.0-1.0,
    "severity": "low|medium|high|critical",
    "fix_steps": ["string"],
    "kb_references": [{ "id": "string", "title": "string", "url": "string" }],
    "past_ticket_refs": [{ "id": "string", "summary": "string", "resolution": "string" }],
    "suggested_group": "string",
    "action": "self_resolve|guided_fix|create_ticket"
  }

5. OCR Pipeline (PaddleOCR)
5.1 Purpose
The OCR pipeline extracts error text from screenshots and screen recordings provided at the kiosk. Extracted text is passed directly to the AI Diagnostic Engine as structured input.

5.2 Pipeline Steps
Input received: screenshot (PNG/JPEG) or screen recording (MP4)
If recording: extract frames at 1 fps using OpenCV, select frame with highest text density
Image preprocessing: resize to 1280px width, convert to grayscale, apply CLAHE contrast enhancement
PaddleOCR (PP-OCRv4) runs inference → returns text blocks with bounding boxes + confidence
Post-processing: filter blocks with confidence <0.7, join text into structured string
Error code extraction: regex pass for known patterns (e.g. 0x000000XX, error XXXX, exit code N)
Structured output passed to Diagnostic Engine as error_text field

5.3 OCR Output Schema
  {
    "raw_text": "string",
    "error_codes": ["string"],
    "confidence_avg": 0.0-1.0,
    "blocks": [{ "text": "string", "bbox": [x,y,w,h], "confidence": 0.0-1.0 }]
  }

5.4 Storage & Retention
Screenshots stored encrypted (AES-256) in /tmp/sessions/{session_id}/
Purged on session close unless attached to a Freshworks ticket
If ticket created: uploaded to Freshworks attachment, then purged locally
Screen recordings: processed in-memory only — never stored to disk

6. RAG Pipeline — KB & Past Ticket Lookup
6.1 Overview
The RAG (Retrieval Augmented Generation) pipeline enables TED to search past resolved Freshworks tickets and KB articles semantically, surface relevant resolutions, and pass them as context to the LLM for fix generation.

6.2 Embedding & Vector Store
Parameter
Specification
Embedding Model
sentence-transformers/all-MiniLM-L6-v2 (384 dimensions)
Vector Store
pgvector extension on PostgreSQL (local, no extra cost)
Similarity Metric
Cosine similarity
Match Threshold
>0.80 = surface result; 0.60–0.80 = include as context only
Top-K Results
Return top 3 matches per query
Index Refresh
Nightly sync from Freshworks (new tickets + KB updates)

6.3 Data Ingestion Pipeline
Nightly cron job calls Freshworks API: fetch resolved tickets (last 90 days) + all KB articles
Each ticket/KB chunked into: title + description + resolution (max 512 tokens)
Embedding generated per chunk via sentence-transformers
Upserted into pgvector table with metadata (ticket ID, category, resolved date)
Stale entries (>180 days, no views) soft-deleted from index

6.4 Vector DB Schema
  CREATE TABLE knowledge_embeddings (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_type VARCHAR(20),   -- 'ticket' | 'kb_article'
    source_id   VARCHAR(100),  -- Freshworks ticket/article ID
    title       TEXT,
    content     TEXT,
    resolution  TEXT,
    embedding   vector(384),
    category    VARCHAR(100),
    created_at  TIMESTAMP DEFAULT now(),
    last_used   TIMESTAMP
  );
  CREATE INDEX ON knowledge_embeddings USING ivfflat (embedding vector_cosine_ops);

7.4 Error Handling & Retry Logic
HTTP 429 (rate limit): exponential backoff — retry at 2s, 4s, 8s (max 3 retries)
HTTP 5xx (server error): retry once after 5 seconds, then fallback to local queue
Offline fallback: ticket payload stored in PostgreSQL queue, synced when connection restored
All API failures logged to audit table with timestamp and error code

8.2 Middleware Stack
JWT validation middleware: validates Bearer token on all protected routes
CORS middleware: restricted to kiosk UI origin only
Rate limiting: 60 requests/minute per session token (slowapi)
Request logging: all requests logged with session ID, endpoint, latency, status
Error handler: standardised error response schema across all routes

8.3 Async Task Handling
Long-running tasks (LLM inference, OCR on recordings) run as background tasks via FastAPI BackgroundTasks or Celery workers:
Diagnosis requests return immediately with a job_id
Client polls GET /diagnose/{id} every 2 seconds until status = complete
Timeout: 30 seconds max — returns partial result with confidence=0 if exceeded

9.3 Offline / Degraded Mode
If FastAPI backend unreachable: show manual ticket form (name, issue description, priority)

Degraded mode banner shown to employee with estimated SD response time
